# AI 能力扩展三种方式对比：技能(Skills)、提示词(Prompt)、知识库(RAG)

## 一、根本区别总览
这三种方式是 AI 扩展能力的三个完全不同维度，核心区别在于**作用阶段、作用范围和执行主体**的不同：

| 对比维度 | Skills（技能） | 提示词（Prompt） | 知识库（RAG/文档阅读） |
|---------|---------------|-----------------|-----------------------|
| **本质** | AI 的"工作流程手册"+"工具扩展包" | AI 的"临时工作指令" | AI 的"外部参考资料库" |
| **作用层级** | 系统层面，长期生效 | 单次对话层面，临时生效 | 知识层面，全局复用 |
| **执行主体** | 系统/AI 自主按流程执行 | AI 按用户指令执行 | AI 参考内容自主执行 |
| **能力边界** | 可扩展 AI 底层能力（调用工具、执行复杂流程） | 仅在 AI 现有能力范围内调整输出 | 仅补充 AI 缺失的知识内容 |
| **复用范围** | 所有对话、所有用户通用 | 当前对话上下文有效 | 所有对话、所有用户通用 |
| **开发成本** | 高（需要代码开发、工具对接、测试） | 极低（自然语言描述即可） | 中等（文档整理、向量化处理） |
| **灵活性** | 低（流程固定，调整需要改代码） | 极高（随时调整要求） | 中等（更新文档即可） |
| **结果稳定性** | 极高（按固定流程执行） | 低（依赖模型理解和推理） | 中等（依赖检索准确性） |

---

## 二、详细工作方式解析
### 1. Skills（技能）：给 AI 装"专业插件"
#### 工作原理
技能是预先编写的**标准化工作流程+工具调用逻辑**，通常包含代码、配置文件、第三方工具集成。相当于给 AI 安装了专门的 APP，让它具备原本没有的能力。技能被触发后，AI 会严格按照预定义的步骤执行，不需要用户每次重复 instructions。

#### 典型架构
```
┌─────────────────┐
│  技能触发规则    │  关键词匹配/意图识别/指令触发
├─────────────────┤
│  流程控制逻辑    │  预定义的执行步骤、分支判断、异常处理
├─────────────────┤
│  工具集成层      │  对接 API、数据库、文件系统、硬件设备
├─────────────────┤
│  结果处理模块    │  格式化输出、错误处理、日志记录
└─────────────────┘
```

#### 实际案例（QQBot 定时提醒技能）
```python
# 技能核心逻辑示例
@skill.on_keyword("提醒我")
async def schedule_reminder(message):
    # 1. 解析时间和内容（NLP 提取实体）
    time_str = extract_time(message.content)
    content = extract_content(message.content)
    
    # 2. 调用系统定时任务接口
    job_id = await cron_service.create_job(
        time=time_str,
        action="send_message",
        params={"user_id": message.user_id, "content": content}
    )
    
    # 3. 返回确认消息
    return f"已设置提醒：{time_str} {content}，任务ID：{job_id}"
```

用户只需要说"明天下午3点提醒我开会"，AI 就自动完成所有操作，不需要用户告诉它"先解析时间，再调用 cron，到点发消息"。

#### 效率特点
✅ **执行效率极高**：复杂流程一次开发，N 次复用，零重复成本  
✅ **能力天花板最高**：可以突破 AI 本身限制，调用外部系统、执行命令、对接硬件  
❌ **开发成本高**：需要开发、测试、部署，适合高频通用场景  
❌ **迭代周期长**：调整流程需要修改代码、重新部署

### 2. 提示词（Prompt）：给 AI 下"临时工作指令"
#### 工作原理
提示词是用户在对话中给 AI 的**单次指令**，仅在当前对话上下文中生效。相当于你临时给员工安排一项具体工作，告诉它这次要按什么标准、什么格式、什么思路完成。完全依赖 AI 本身的理解能力和现有知识，不会扩展底层能力。

#### 常见类型
1. **角色设定类**："你是一个资深产品经理，擅长写 PRD 文档"
2. **输出格式类**："把结果整理成 Markdown 表格，包含名称、价格、优缺点三列"
3. **思考方式类**："一步步分析这个问题，先讲优势，再讲风险，最后给建议"
4. **约束条件类**："回答控制在 100 字以内，不要用专业术语"

#### 实际案例
```
用户输入：
"""
请把下面这段产品介绍改成短视频文案：
产品：智能保温杯
特点：24小时保温、提醒喝水、APP 连接、健康数据统计
要求：风格活泼，用网络热词，长度控制在 80 字以内，最后加一个话题标签
"""
```
AI 会严格按照这些要求输出，但下一次对话不重复说明的话，它不会自动按这个风格生成。

#### 效率特点
✅ **使用成本极低**：想到什么直接说，零开发成本  
✅ **灵活性最高**：随时调整要求，适配不同场景和个性化需求  
❌ **重复成本高**：相同要求每次对话都要重新说一遍  
❌ **稳定性差**：同样的提示词不同时间输出可能不一致，依赖模型能力  
❌ **能力有限**：只能调整输出，不能让 AI 做本身不会的事（比如调用 API、查数据库）

---

### 3. 知识库（RAG/文档阅读）：给 AI 配"专属参考书"
#### 工作原理
知识库是把 AI 训练数据里没有的**专有知识**（公司内部文档、产品手册、项目资料、法律法规等）提前做向量化处理，存储在向量数据库中。当用户问相关问题时，AI 先检索知识库中最相关的内容，再结合检索到的信息生成回答，相当于给 AI 提供了实时查阅的参考资料。

#### 工作流程
```mermaid
graph LR
A[用户提问] --> B[问题向量化]
B --> C[向量数据库检索相似内容]
C --> D[相关内容拼接成上下文]
D --> E[LLM 结合上下文生成回答]
E --> F[返回给用户]
```

#### 实际案例
```
预处理阶段：
1. 上传公司所有产品手册、售后政策、常见问题到知识库
2. 系统自动对文档做切片、向量化、存储到向量库

用户提问："产品A保修期多久？非人为损坏可以免费维修吗？"
执行阶段：
1. 检索知识库找到"售后政策.md"中相关段落
2. 把问题+相关段落一起发给 LLM
3. LLM 基于真实内容生成准确回答
```

如果没有知识库，AI 会根据训练数据里的通用知识回答，很可能出现编造内容的情况。

#### 效率特点
✅ **知识扩展能力强**：可以让 AI 掌握任何专有知识，不需要重新训练模型  
✅ **复用性高**：文档上传一次，所有用户、所有对话都可以使用  
✅ **准确性高**：基于真实文档回答，减少幻觉  
❌ **能力边界有限**：只能补充知识，不能让 AI 学会新技能或调整输出风格  
❌ **有检索误差**：文档太多时可能检索到不相关内容，影响回答准确率  
❌ **维护成本**：文档更新需要重新同步到知识库

## 三、效率对比与适用场景
### 1. 效率对比表
| 场景 | 技能 | 提示词 | 知识库 |
|------|------|--------|--------|
| 每天自动发送天气日报 | 🚀 10/10（一次开发，自动执行） | 🟡 3/10（每天都要手动问） | ❌ 0/10（完全不适用） |
| 写一份特定风格的产品文案 | 🟡 4/10（开发成本高，不划算） | 🚀 10/10（直接写要求就行） | ❌ 0/10（完全不适用） |
| 回答公司员工福利相关问题 | 🟡 5/10（需要对接 HR 系统） | 🟡 5/10（每次都要贴政策） | 🚀 10/10（上传政策文档即可） |
| 智能客服自动处理售后问题 | 🚀 9/10（对接工单系统+订单查询） | 🟡 3/10（每次都要给规则） | 🚀 9/10（产品知识+售后政策） |
| 代码助手自动调试代码 | 🚀 10/10（对接 Git+CI/CD+调试工具） | 🟡 4/10（每次都要说明规则） | 🟡 6/10（代码库+文档） |

### 2. 适用场景总结
#### ✅ 优先用技能的场景
- 高频、重复、流程固定的任务
- 需要调用外部工具/系统/API 的任务
- 需要自动执行、不需要用户干预的任务
- 对结果准确性、稳定性要求极高的任务

#### ✅ 优先用提示词的场景
- 一次性、个性化、临时的任务
- 调整输出风格、格式、思考方式
- 简单的逻辑推理、内容生成
- 快速验证想法、原型

#### ✅ 优先用知识库的场景
- 需要基于专有知识、内部文档回答的问题
- 避免 AI 产生幻觉、要求回答有事实依据
- 知识更新频繁，不需要重新训练模型
- 企业内部知识库、客服问答、产品手册查询

## 四、三者配合使用的最佳实践
复杂场景通常需要三者结合使用，才能达到最佳效果：

### 案例：企业智能客服系统
```
┌─────────────────┐
│  技能层          │ 对接订单系统、物流查询、工单创建、退款接口
├─────────────────┤
│  提示词层        │ 要求客服回答热情、使用统一话术、引导用户好评
├─────────────────┤
│  知识层          │ 产品手册、售后政策、常见问题、活动规则
└─────────────────┘
```
用户问："我买的产品还没收到，能帮我查一下吗？"
1. **技能**：自动调用订单查询接口，获取订单状态和物流信息
2. **知识库**：查询物流异常处理政策
3. **提示词**：按照客服话术要求，把信息整理成友好的回答

### 案例：AI 代码助手
```
┌─────────────────┐
│  技能层          │ 对接 Git、CI/CD、测试工具、部署系统
├─────────────────┤
│  提示词层        │ 要求代码符合公司规范、加注释、写单元测试
├─────────────────┤
│  知识层          │ 代码库文档、架构设计、API 文档、历史 Bug 记录
└─────────────────┘
```
用户说："帮我看看这个接口为什么报错？"
1. **技能**：自动拉取代码、查看日志、运行调试工具
2. **知识库**：查询相关接口文档、历史类似问题的解决方案
3. **提示词**：按照规范给出错误原因和修复方案，包含代码示例

### 通用配合原则
1. **技能做能力底座**：所有需要工具调用、流程化的操作都做成技能
2. **提示词做灵活调整**：针对不同场景、不同用户的个性化要求用提示词调整
3. **知识库做事实依据**：所有需要专有知识、事实性内容都存在知识库
4. **边界清晰**：不要用提示词实现复杂流程，不要用技能处理个性化需求，不要用知识库回答非事实性问题

## 五、常见误区
### ❌ 误区1：用提示词实现复杂流程
比如试图用长提示词让 AI 自动完成"查询订单→判断是否符合退款条件→创建退款工单→通知用户"的流程，结果就是不稳定、容易出错、无法处理异常。**复杂流程一定要用技能实现**。

### ❌ 误区2：把技能做的太灵活
试图在技能里支持各种个性化参数、自定义格式，结果就是技能越来越复杂、维护成本极高。**技能只做通用流程，个性化调整交给提示词**。

### ❌ 误区3：把所有内容都塞到知识库
不管是流程、规则、还是知识全部丢到知识库，结果就是检索准确率下降、回答变慢、出现冲突。**知识库只存事实性知识，流程规则做成技能，临时要求用提示词**。

### ❌ 误区4：三者互相替代
觉得"提示词写得好就不需要技能"、"有了知识库就不用提示词"，实际上三者是互补关系，没有谁能替代谁，配合使用才能发挥最大价值。

## 六、选型决策树
```
是否需要调用外部工具/系统？
├─ 是 → 用技能实现
└─ 否 → 是否需要基于专有知识回答？
   ├─ 是 → 是否是高频通用问题？
   │  ├─ 是 → 存入知识库
   │  └─ 否 → 直接把信息贴在提示词里
   └─ 否 → 是否需要个性化/特殊格式输出？
      ├─ 是 → 用提示词控制
      └─ 否 → 直接让 AI 回答
```
